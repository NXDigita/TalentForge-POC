## WEEK 2 — Sandbox Execution & Auto-Grading (Jul 20–24) ⚠️ Critical Path

| Day | Rohan — Backend / Sandbox / Infra | Karthikeyan — Frontend / Tools | Done When |
|---|---|---|---|
| **Day 6** (Mon Jul 20) | • Build runner images `worker/images/`: `Dockerfile.python` (python:3.11-slim + pylint), `Dockerfile.node` (node:18-alpine + eslint), `Dockerfile.java` (eclipse-temurin:17 + checkstyle jar)<br>• Each image: non-root user `runner` (uid 1000), entrypoint `run.sh` reading code from `/box/solution.*`<br>• Execution via `dockerode`: `HostConfig: { NetworkMode:'none', Memory: 268435456, ReadonlyRootfs:true, Tmpfs:{'/tmp':'rw,size=50m'}, PidsLimit:64, CpuShares:512 }`<br>• Hard kill timer: `setTimeout(30_000)` → `container.kill()` → status TIMEOUT | • Run/Submit UX: submit → button morphs to spinner "Queued…" → socket status chips (Queued → Running → Graded)<br>• `useGradingSocket(submissionId)` hook: joins room `submission:{id}`, updates TanStack cache on events<br>• Results panel shell `src/components/ResultsPanel.tsx` under editor (collapsible) | • `docker run` of each image executes hello-world in all 3 langs<br>• Network disabled verified: `curl` inside container fails<br>• 30s infinite loop gets killed, job marked TIMEOUT |
| **Day 7** (Tue Jul 21) | • Correctness runner in `worker/src/grader/correctness.ts`: for each test case pipe `stdin`, capture stdout/stderr/exitCode, `trim()` diff vs `expectedStdout`, weighted pass % → correctness score (0–100)<br>• Persist per-case results to `Submission.scores.cases[]`<br>• Java compile step: `javac Solution.java` first; compile errors → status COMPILE_ERROR with stderr | • Test-case results table: pass/fail icon per case, expected vs actual diff view (`react-diff-viewer-continued`), exec time + memory per case<br>• Score ring component (SVG donut, animated 0→score)<br>• Status states UI: COMPILE_ERROR (show stderr in mono block), TIMEOUT, OOM | • Submitting correct Two Sum in all 3 langs → 100 correctness<br>• Wrong solution shows exact failing cases with diff<br>• Compile error renders stderr readably |
| **Day 8** (Wed Jul 22) | • Complexity heuristic `grader/complexity.ts`: run solution at input sizes n, 2n, 4n (generator per problem), fit time ratios → classify O(n)/O(n log n)/O(n²) → complexity score vs problem's `expectedComplexity`<br>• Style score `grader/style.ts`: run pylint/eslint/checkstyle inside container, map warnings/errors → 0–100<br>• Composite: `total = 0.6*correctness + 0.3*complexity + 0.1*style`; persist + emit in `grading:complete` | • Feedback panel: three score bars (correctness/complexity/style) + composite<br>• LLM feedback formatter: `POST /api/feedback/format` (backend proxy → Claude API) converts raw grader JSON into 3 coaching bullets; render with typing effect; behind feature flag `VITE_FF_LLM_FEEDBACK` | • O(n²) Two Sum scores lower on complexity than O(n) version<br>• Style violations reduce score visibly<br>• LLM feedback returns 3 actionable bullets in <3s (or flag off cleanly) |
| **Day 9** (Thu Jul 23) | • Static pre-check `grader/precheck.ts` regex/AST blocklist: py `subprocess|os\.system|eval\(|__import__`, js `child_process|require\(['"]fs`, java `Runtime\.getRuntime|ProcessBuilder` → reject with status BLOCKED before container run<br>• Graceful failure: worker `try/catch` all paths, `job.moveToFailed` never thrown upward, queue never stalls (`stalledInterval` tuned)<br>• Retry policy: infra errors retry ×2; user errors (compile/timeout) never retry | • Submission History page `src/pages/History.tsx`: table of attempts (date, language, score, status), row click → past result detail<br>• Score trend sparkline per problem (Recharts `LineChart`)<br>• Resubmit flow: cooldown countdown chip (server returns `nextAllowedAt`), disable submit until then | • Malicious snippets (fork bomb, fs read, network call) rejected pre-execution with clear message<br>• Killing worker mid-job → job recovers, no zombie containers (`docker ps` clean)<br>• History paginates and renders past scores |
| **Day 10** (Fri Jul 24) | • **Joint E2E:** run 10 canned solutions (correct/wrong/slow/malicious/infinite-loop) × 3 languages, fix all failures<br>• Load test: `artillery` script, 20 concurrent submissions → p95 grading < 5s; tune worker `concurrency`, pre-pull images, keep warm container pool (optional)<br>• Sentry: `@sentry/node` in backend + worker, source maps uploaded | • **Joint E2E** from browser side; record every UX papercut, fix top 10<br>• Sentry `@sentry/react` + session replay on errors<br>• Empty/error states for results panel (queue down, socket disconnect → auto-reconnect + refetch fallback poll every 5s) | • **Week 2 exit:** submit → live score < 5s p95, all 3 languages, on staging<br>• 20 concurrent submissions, 0 failures<br>• Sentry captures a forced test error from both apps |
